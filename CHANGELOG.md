# Changelog

## [Unreleased]

### Added
- Terminal sessions (SPEC §4): a session can now be a plain shell (`$SHELL`) with no coding agent and no account, launched from a new terminal button in the sessions sidebar (and the collapsed rail). It reuses the new-session dialog's branch machinery — but with **use separate branch off by default** — spawns a shell in the worktree, and is kept alive like any agent session (a daemon restart interrupts it; resume relaunches a fresh shell in the same worktree). It shows in the sidebar with a **blue** status dot instead of green when running and no account line. `POST /api/sessions` gains an optional `kind` ('agent' | 'terminal') and makes `account_id` optional (required for agents, forbidden for terminals); `sessions.account_id`/`agent_type` are now nullable and a `kind` column is added (migration 009). New `--status-terminal` design token. **Protocol major bump 4.7 → 5.0** (`account_id`/`agent_type` on the session shape changed from required to nullable).
- The collapsed sessions rail shows one clickable status dot per live session beneath its controls, so you can switch sessions without reopening the sidebar.
- A session that joins an already-existing shared worktree (`separate_branch: false`, where other agents are working in the same directory) now opens with a short concurrency heads-up prefixed to its prompt — expect the working tree to shift underneath you; avoid resets, force-pushes, or branch deletion that would trample the others. It still skips the onboarding preamble (the environment already exists); SPEC §4. No protocol change.
- The right sidebar's session rows gain iconography: a git-branch icon in front of the branch name, and the account line's leading glyph is now the coding agent's own brand mark (Claude, Codex, …) rather than a generic person icon, falling back to the person glyph for an unknown agent. New `components/agent-icon.tsx` keyed off `sessions.agent_type`. No protocol change.
- Accounts can be renamed in Settings → Accounts: the name is an inline editable field that saves when it loses focus (or on Enter; Escape reverts). Only the display label changes — the account's config dir keeps its original path, so macOS keychain OAuth (bound to that path) is not broken — and renaming to a name already used by another account of the same agent in the profile is a 409. `PATCH /api/accounts/:id` gains an optional `label` (additive, within the 4.6 bump below).
- Tier-1 session migration (SPEC §5/§6, Workstream S): `POST /api/sessions/:id/migrate {account_id}` moves a session to another account of the same profile and agent and resumes it there. Because the conversation already lives in the profile's shared store (Task 17), nothing on disk moves — the daemon kills any live process, repoints `sessions.account_id`, and runs the normal resume path under the target's credentials (same worktree, same branch, same `agent_session_ref`), recording a `migrated` event. Validations fail closed in order: unknown target `404`, cross-profile `400 cross_profile_account`, wrong agent `400 agent_mismatch`, current account `400 same_account`, archived `409 session_archived`, logged-out target `409 account_logged_out`, no way to carry the conversation `409 migration_unsupported`. `skip_permissions` is re-evaluated for the target at resume time (§11.4) — a since-closed gate silently drops the flag and says so in the terminal. The workspace session menu gains a **Move to account…** submenu (the profile's same-agent accounts; current disabled; hidden when there is no other target) behind a confirm dialog. The conversation's per-account `todos/` do not travel. Protocol minor bump 4.4 → 4.5 (`migrateSessionRequestSchema`). Acceptance: `docs/acceptance/tier1-migration.md`.
- Ports strip in the workspace (SPEC §9, Phase 5): a slim mono row under the active session's terminal (terminal view only) lists its detected listening ports, polling `GET /api/sessions/:id/ports` every 5s while the session is `running`/`waiting_input` — no manual refresh. Each port is a chip (tooltip: `command · pid <pid>`) opening a menu with all three access paths, always shown since the daemon can't yet tell whether the browser is behind a tunnel: **Open localhost** (`http://localhost:<port>`), **Open via proxy** (the Task 15 `/proxy/:sid/:port/` cookie bootstrap), and **Copy ssh -L** (`ssh -L <port>:127.0.0.1:<port> user@host` from `GET /api/host`). No protocol change.
- Tier-2 reverse proxy (SPEC §2/§9, Phase 5): `/proxy/:sid/:port/*` forwards to a dev server listening inside that session's process tree, over the already-open SSH tunnel — including the WebSocket upgrade HMR needs. HTTP is forwarded via raw `node:http` (streaming both ways for SSE/chunked), preserving the request path byte-for-byte, rewriting `Host` to `127.0.0.1:<port>` (Vite `allowedHosts`-friendly), dropping hop-by-hop headers, and stripping only the `puddle_proxy` cookie; a connect failure is a `502` while the upstream's own status passes through unchanged. The bare `/proxy/:sid/:port` 302-redirects to the trailing-slash form so relative asset URLs resolve, and an unknown port triggers one fresh re-scan before a `403`. No protocol change (`/proxy` is documented in SPEC, not a JSON API shape).
- `GET /api/sessions/:id/ports` (SPEC §9, Phase 5 groundwork): lists TCP ports currently listening within a session's process tree (the agent plus every live shell terminal on its stream, and their descendants), detected via `lsof -iTCP -sTCP:LISTEN` on macOS or `ss -tlnp` on Linux and scoped by a `ps`-derived process-tree walk so an unrelated host process never appears. Results are cached per session for 2s with in-flight dedupe. Protocol minor bump 4.3 → 4.4.
- Open-in-editor deep links (SPEC §7): a session's menu gains "Open in VS Code" / "Open in Cursor", each opening the worktree via `vscode://file<path>` / `cursor://file<path>` locally, or the `<scheme>://vscode-remote/ssh-remote+<host><path>` form when an SSH host is known — from the new client setting (Settings → Terminal & editor → "SSH host for editor links") or a `?host=` boot param the CLI will send from Phase 6 onward (captured once, stored per browser, stripped from the address bar). No protocol change.
- Terminal links (SPEC §7): URLs an agent prints are now clickable (via `@xterm/addon-web-links`), opening in a new tab on plain or cmd/ctrl+click. Separately, file paths the agent mentions — `src/foo.ts:12:3`, `./a/b.py`, `/wt/main.c` — underline only after the daemon's `/resolve` confirms the file exists (so prose that merely looks path-shaped stays quiet), and cmd/ctrl+click opens them in the Monaco editor at the given line/column while a plain click keeps terminal selection. Validation is cached per session with short TTLs and a concurrency cap; login terminals get URL links but no file links (no worktree). No protocol change.
- `GET /api/worktrees/:sid/resolve?path=&line=` validates a terminal file-path link before the UI underlines it (SPEC §7, Phase 4 groundwork): relative paths resolve against the worktree root, absolute paths are accepted only when inside it, and both an escape attempt and a missing file collapse to the same non-informative 404 so the client only ever learns "don't underline". Protocol minor bump 4.2 → 4.3.
- History navigator in the workspace (SPEC §8): the left sidebar's **History** mode lists the session's commit log (paginated, newest first); clicking a commit expands its changed files inline, and clicking a file opens its `sha^ → sha` diff as a read-only centre-editor tab. Every diff here is `sha`→`sha^`, so unlike the Diff tab it never touches the shared editor buffer: both sides are private Monaco models created and disposed with the tab, and the root commit's files always render as `added` rather than fetching a nonexistent parent. Absolute commit dates render in a fixed UTC, not the viewer's local zone, for deterministic output.
- Diff navigator in the workspace (SPEC §8): the left sidebar's **Diff** mode lists the working tree's changed files against its merge-base (polled every 10 s and on focus, no manual refresh); clicking a file opens its diff as a centre-editor tab — added files as a plain editor, deleted files read-only, modified/renamed as a side-by-side diff whose right-hand side IS the editor tab's shared buffer, so ⌘S in the diff saves through the very same conflict-safe path and the agent sees the change.
- Monaco editor in the workspace (SPEC §8): opening a file from the explorer adds a tab to a new editor pane above the terminals (a resizable vertical split), with conflict-safe saves and browser-cached drafts. ⌘S writes back with an optimistic-concurrency guard — if the file changed on disk (usually the agent) the save offers **Reload** or **Overwrite** rather than clobbering. Unsaved edits are restored on reopen from the IndexedDB draft, a second window's save/edit surfaces as a passive badge, and binary or too-large files fall back to a Download panel. Monaco is fully self-hosted (no CDN). No protocol change.
- File explorer panel (SPEC §8): a resizable third panel in the workspace showing one worktree's file tree, bound to the active session tab by default with a pin control to lock it to a chosen session instead. Directories expand lazily; rows support drag-and-drop and clipboard-paste upload (folders are rejected with a toast — zip them first) and a context menu for **Download** (file or zipped directory) and **Copy path**. New `src/lib/worktree-queries.ts` TanStack Query hooks (`useWorktreeTree`, `useWorktreeFile`, `useSaveWorktreeFile`, `useWorktreeDiff`, `useFileAt`, `useWorktreeLog`, `useCommitShow`) plus `uploadFiles`/`downloadPath`, and an owned `components/ui/context-menu.tsx`. No protocol change.
- Unsaved editor buffers are cached in the browser's IndexedDB (`puddle-drafts`, debounced ~1 s) so they survive a full browser restart — per-browser only, never synced through the daemon (SPEC §11).
- Worktree file API family: browse a session's worktree one directory level at a time, read/write text files with optimistic-concurrency guards, drag-in multipart upload, and single-file/zipped-directory download (`GET/PUT /api/worktrees/:sid/tree|file`, `POST /upload`, `GET /download`) — the daemon half of the Phase 3 file explorer, path-contained against traversal and symlink escape.
- Read-only git inspection API for a session's worktree — diff against the merge-base or an explicit commit, a file's content at any ref, paginated commit log, and a single commit's detail (`GET /api/worktrees/:sid/diff|file-at|log|show/:sha`) — plus an `ahead`/`behind`/`dirty_files` `git_summary` on `GET /api/sessions/:id`, the daemon half of the Phase 3 history view. `isoTimestamp` now accepts a numeric UTC offset (previously Z-only) since git's `%aI` renders the commit author's own offset verbatim rather than normalising to Z; every daemon-produced timestamp is still always Z, so this only widens what a new `authored_at` field is allowed to carry.
- Shared protocol schemas for worktree file browsing/editing and git inspection (tree/file/upload/download, diff/file-at/log/show), a session `git_summary` field, and two new `ui_state` keys (`active_editor_tab`, `explorer_open`) — groundwork for the Phase 3 file explorer and history view; no endpoints yet. Protocol minor bump 4.1 → 4.2.

### Fixed
- Opening a file in the built app no longer blanks the screen: the CSS minifier shortens `#ffffff` to `#fff` inside the bundled theme tokens, which monaco's token parser rejects ("Illegal value for token color"); the runtime token reader now expands shorthand hex back to canonical form.
- Closing the active session tab no longer leaves a zombie header: the tab is removed cleanly instead of being resurrected by the deep-link effect during the render where the URL param still lagged behind the close.
- Renaming a session from the UI now sticks and shows immediately. The `.puddle/session-title` watcher re-applied the file's content on _every_ `.puddle` change (a pasted image, a notes edit), silently overwriting a title set from the UI; it now reacts only to genuine edits of the file, so a manual rename survives unrelated worktree activity. The rename also broadcasts live (new `renamed` WS message) instead of only updating the client that issued it.
- Sessions can no longer be hijacked by an agent's in-TUI login screen: puddle verifies the account's login with the agent itself (`checkLoggedIn`) before every create and resume — a logged-out account is a clear 409 `account_logged_out` pointing at Settings → Accounts, and stored logged-in badges are re-verified at boot. This matters because macOS keychain OAuth is bound to the config-dir path, so migration 004's directory rename silently logged existing accounts out (a one-time re-login rebinds them; ids never change again).
- Resume survives a discarded session id: when the recorded conversation ref matches nothing, the daemon recovers the conversation by the working directory recorded in its JSONL (`discoverSessionRef`), updates the ref, and resumes — with a `session_ref_recovered` event in the audit trail.
- claude-code sessions no longer demand a second sign-in: `claude auth login` authenticates but leaves first-run onboarding incomplete, so the first session ran Claude Code's setup wizard (theme → sign-in again → trust) — which also discarded the preset `--session-id` and broke resume. Fresh account config dirs are now seeded with onboarding marked complete (new adapter hook `prepareConfigDir`, verified against Claude Code 2.1.208).
- Resuming a session whose conversation file is gone (or was never recorded under the expected id) returns a clear 409 `conversation_missing` instead of spawning an agent that immediately dies with "No conversation found" (new adapter hook `hasConversation`).

### Security
- The reverse proxy never forwards the `Authorization` header to an upstream dev server (SPEC §9): a client that authenticates with `Authorization: Bearer <daemon-token>` would otherwise hand the full-RCE daemon token to a session's (potentially agent-generated) dev server. Both the HTTP forward and the raw WebSocket upgrade now drop `authorization` alongside the existing `puddle_proxy` cookie and `puddle_token` query strips; the query strip also decodes each param name first, so a percent-encoded `puddle%5Ftoken=` (which authenticates via WHATWG decoding) can no longer survive into the forward. Trade-off: an upstream needing its own `Authorization` does not receive one — its auth must ride a different header.
- `/proxy` auth (SPEC §2): a browser tab (and the WebSocket handshake it opens) can't send a bearer header, so the proxy also accepts a `puddle_proxy` cookie and a one-shot `?puddle_token=` query param. A `?puddle_token=` GET plants `Set-Cookie: puddle_proxy=<token>; Path=/proxy; HttpOnly; SameSite=Lax` and 302-redirects with **only** that param stripped, so the token never lingers in the address bar and every later request (cookie-carried) is authed — including the un-headerable WS upgrade. The cookie value is the daemon token itself (a minted second secret would add state without moving a trust boundary). All comparisons are timing-safe, Host/Origin validation guards `/proxy` on both the HTTP and raw WS-upgrade paths, and the `puddle_token` query pair is always spliced out of the forwarded request line so the daemon token never reaches an upstream dev server's access logs. Documented caveat: a proxied app shares the daemon's origin, so a hostile dev server could read the UI's stored token — accepted for v1 on a single-trusted-user box.
